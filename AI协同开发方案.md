## 3. 技术规范

### 3.1 代码规范

#### 前端规范
- **命名规范**：
  - 组件名：PascalCase（如 `AuthView.vue`）
  - 变量名：camelCase（如 `loginForm`）
  - 常量：UPPER_SNAKE_CASE（如 `API_BASE_URL`）
- **文件结构**：按功能模块组织
- **代码风格**：使用 Prettier 格式化
- **注释**：关键逻辑添加注释

#### 后端规范
- **命名规范**：
  - 类名：PascalCase（如 `AuthController`）
  - 方法名：camelCase（如 `register`）
  - 变量名：camelCase（如 `userService`）
- **包结构**：按分层架构组织
- **代码风格**：遵循阿里巴巴Java开发手册
- **注释**：方法添加Javadoc

### 3.2 目录结构规范

### 3.3 API 规范

- **接口路径**：`/api/{模块}/{功能}`
- **HTTP方法**：
  - GET：查询
  - POST：创建
  - PUT：更新
  - DELETE：删除
- **响应格式**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {}
  }
  ```
- **错误处理**：统一异常处理，返回错误码和错误信息

## 4. 开发流程

### 4.1 分支管理

- **main**：主分支，用于发布
- **develop**：开发分支，集成所有功能
- **feature/{模块}**：功能分支，如 `feature/activity`
- **bugfix/{问题}**：bug修复分支

### 4.2 开发步骤

1. **需求分析**：理解功能需求
2. **技术设计**：设计API接口和数据结构
3. **代码实现**：
   - 后端：实现接口和业务逻辑
   - 前端：实现页面和交互
4. **单元测试**：编写测试用例
5. **集成测试**：测试前后端集成
6. **代码审查**：互相审查代码
7. **合并分支**：将功能分支合并到develop
8. **发布**：从develop合并到main

### 4.3 协同工具

- **代码仓库**：GitHub
- **任务管理**：GitHub Issues
- **代码审查**：GitHub Pull Requests
- **沟通工具**：根据团队偏好选择

## 5. 代码管理

### 5.1 Git 提交规范

使用 Conventional Commits 规范：

- **feat**：新功能
- **fix**：修复bug
- **docs**：文档更新
- **style**：代码格式调整
- **refactor**：重构
- **test**：测试相关
- **chore**：构建/工具相关

### 5.2 代码审查

- **Pull Request**：每个功能分支必须通过代码审查
- **审查要点**：
  - 代码质量
  - 安全性
  - 性能
  - 可读性
  - 测试覆盖

### 5.3 版本管理

- **版本号**：语义化版本（如 1.0.0）
- **发布流程**：
  1. 合并所有功能到develop
  2. 测试通过后合并到main
  3. 打标签发布

## 6. 测试策略

### 6.1 单元测试

- **前端**：使用 Vitest
- **后端**：使用 JUnit 5
- **覆盖率**：目标 80% 以上

### 6.2 集成测试

- **API测试**：使用 Postman 或 Swagger
- **端到端测试**：使用 Cypress

### 6.3 性能测试

- **前端**：页面加载时间
- **后端**：接口响应时间
- **数据库**：查询性能

## 7. 部署方案

### 7.1 开发环境

- **前端**：`npm run dev`（http://localhost:3000）
- **后端**：`mvn spring-boot:run`（http://localhost:8080）
- **数据库**：本地 MySQL

### 7.2 生产环境

- **前端**：
  1. `npm run build` 构建
  2. 部署到 Nginx
- **后端**：
  1. `mvn clean package` 打包
  2. 部署到服务器
- **数据库**：生产环境 MySQL

### 7.3 容器化（可选）

- **Docker**：使用 Docker Compose 部署
- **CI/CD**：使用 GitHub Actions 自动部署
